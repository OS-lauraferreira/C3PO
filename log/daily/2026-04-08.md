## M2.2 Framework Pivot — Key Decisions

**Context:** Meeting with Vera about Reviewers Framework revealed misalignment. I was focused on reviewer team structure/reorganization. Vera's feedback: we don't need to restructure teams, we need to standardize the review process.

**Problem Reframed:**
- ❌ **Old focus:** Who reviews? Reviewer tiers, authority levels, team structure
- ✅ **New focus:** How to review? What questions must reviewers ask before approving?

**Core Insight:** CAB Reviewers lack a structured way to evaluate changes. They don't know what information to look for or what questions to ask. This causes:
- Inconsistent decisions (same change reviewed differently by different reviewers)
- Approval bottlenecks (reviewers must ask follow-up questions)
- Low-quality RFCs (requesters don't know what reviewers need)

**Solution:** Review Questions Framework
- Define standardized checklists by change type (Emergency/Normal/Standard)
- Core questions (Q1-Q6) for all changes
- Type-specific questions (E1-E6, N1-N8, S1-S5)
- Clear decision criteria: approve / request modifications / reject

**Connection to M2.1:** 26 Operation Types analysis (6 Standard + 20 Normal) maps directly to review criteria:
- Standard types: Verify catalog compliance (S1-S5)
- Normal types: Detailed review (N1-N8)
- Pilot with top 3 Normal operation types by RFC volume

**Next Steps:**
1. Complete review questions definition (Emergency/Normal/Standard checklists)
2. Identify top 3 Normal operation types from M2.1 analysis (by RFC volume)
3. Prepare presentation for Vera (April 16) with pilot plan
4. Pilot framework on 15 RFCs (5 per top operation type)
5. Refine based on feedback, then full rollout
