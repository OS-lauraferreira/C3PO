## Executive Summary

Analysis of all 2025 RFCs reveals a **single canonical set of 32 unique Operation Types** that covers both Standard and Normal changes. This is NOT two separate lists, but one unified taxonomy with different applicability levels.

**Key Insight:** Standard and Normal changes are **subsets** of the same 32 canonical types, not parallel taxonomies.

---

## The Canonical Model

### Single Source of Truth: 32 Operation Types (Canonical)

A unified set of 32 operation types that represents all change patterns observed in 2025:

| Category | Count | Definition |
|---|---|---|
| **Total Canonical Types** | **32** | Complete, unique set covering all observed operations |
| **Standard applicability** | 22 types | Subset approved for self-service (18 current + 4 new) |
| **Normal applicability** | 26 types | Subset requiring CAB review (18 overlap + 8 unique) |
| **Overlap (both Standard & Normal)** | 18 types | Operations usable in either context depending on change scope |

---

## What This Means

### Before (Incorrect Model)
```
Standard Changes: 22 types (isolated)
Normal Changes: 26 types (isolated)
= 48 types total (duplicated thinking)
```

### After (Correct Model — Canonical)
```
Canonical Types: 32 unique types
├── Usable as Standard (22 types)
├── Usable as Normal (26 types)
└── Overlap (18 types — context-dependent routing)
```

**The overlap (18 types) means:** The same operation can be Standard OR Normal depending on scope, risk, or environment.

---

## Breakdown: Standard Changes (22 Types)

**Composition:**
- **18 current types:** Already in use, validated, proven safe for self-service
- **4 new/recommended types:** Analysis identified operations currently classified as Normal that should be promoted to Standard (high success rate, low risk profile)

**Implication:** Standard operations are a **proven subset** of the 32 canonical types. No new operations introduced; just reclassification based on 2025 data.

**ODC-CAT Impact:** Create 22 Catalog articles for Standard operations (pre-approved, self-service tier).

---

## Breakdown: Normal Changes (26 Types)

**Composition:**
- **18 types with overlap:** Same operations usable as Standard in some contexts, but require CAB review in others (higher scope, more systems affected, cross-ring changes, etc.)
- **8 unique types:** Only ever appear as Normal; too risky for self-service (database changes, security patches, infrastructure reconfigurations, etc.)

**Implication:** Normal operations include both reclassifiable types AND inherently high-risk types.

**ODC-CAT Impact:** Create 26 Catalog articles for Normal operations (CAB review required, but possible auto-escalation to Emergency for edge cases).

---

## The Overlap: Why 18 Types Exist in Both

| Operation Type Example | Standard Use Case | Normal Use Case |
|---|---|---|
| "App configuration update" | Single app, single ring, rollback < 5 min | Multi-ring, cross-team, rollback > 1 hour |
| "Service restart" | Single instance, testing ring | Production ring with traffic, coordinated restart |
| "Feature flag toggle" | Known-safe flag, gradual rollout | New flag, immediate 100% rollout, no gradual test |
| ... | Low risk, bounded scope | High risk, broad scope |

**Pattern:** The same operation type can be Standard OR Normal depending on **scope, scale, and risk context**.

**Governance Rule:** RFC submission must clearly state scope. System auto-classifies based on:
1. Operation type selected
2. Scope metadata (rings affected, teams involved, estimated downtime)
3. Risk assessment algorithm → routes to Standard tier OR Normal tier

## Impact on ODC Catalog Architecture

### Current Thinking (Broken)
- Create 22 Standard articles + 26 Normal articles = 48 articles
- Duplicated logic, confusing for teams
- Maintenance nightmare (change one → must change both)

### Correct Model (Unified)
- Create 32 canonical Catalog articles
- Each article includes: "Standard conditions" + "Normal conditions" + "Emergency escalation rules"
- Teams select ONE operation type; system applies context-based routing

**Example Catalog Article:**

```
[ODC-CAT-001] Service Restart

Standard Conditions:
├─ Single instance
├─ Testing/Staging rings only
├─ Estimated downtime < 5 minutes
└─ Rollback: Manual restart

Normal Conditions:
├─ Multiple instances
├─ Production ring
├─ Estimated downtime 5–30 minutes
├─ Coordinated with team leads
└─ Rollback procedure required

Emergency Escalation:
├─ Production ring + customer-facing service
├─ Unplanned/reactive restart
└─ Auto-escalate to Tier 3
```

## Duplicate Runbooks Analysis

**Finding:** Some operations have multiple runbooks (legacy + current implementations).

**Resolution Strategy:**
- **Legitimate variants:** Keep separate articles; document when each applies
- **Legacy debt:** Deprecate old runbook; migrate to canonical version

**Action:** Consolidate runbooks during ODC-CAT creation (map each 32 canonical type to ONE current runbook; deprecate alternatives).
