# Change Review Process Framework — M2.2 (Transform Change Management)

## Overview

The Change Review Process Framework standardizes how CAB Reviewers evaluate RFCs by defining clear review criteria, decision questions, and approval gates for each change type. This ensures consistent, thorough reviews while maintaining segregation of duties and reducing approval bottlenecks.

**Strategic Goal:** Reduce approval bottlenecks and change failure rates by giving reviewers a structured decision framework with clear questions to ask before approving any change.

---

## Problem Statement (Reframed)

### Current State
- **No structured review process** — CAB Reviewers lack a consistent framework for evaluating changes
- **Unclear approval criteria** — Reviewers don't know what questions to ask or what information is needed to approve
- **Inconsistent decisions** — Same change type reviewed differently by different reviewers
- **Missing information in RFCs** — Requesters don't know what reviewers need to see
- **Segregation of duties enforced but not optimized** — Governance control exists but review quality varies

### Impact
- **Approval bottlenecks** — Reviewers must ask follow-up questions, delaying approval
- **Inconsistent risk assessment** — Decisions vary by reviewer, creating unpredictability
- **Low-quality RFCs** — Requesters submit incomplete information
- **Change Failure Rate (CFR)** — Poor review quality allows risky changes through
- **Reviewer frustration** — No clear authority on what constitutes "approval-ready"

---

## Proposed Solution: Structured Review Questions Framework

Instead of focusing on *who* reviews (already governed by segregation of duties), focus on *what* reviewers must validate before approval.

**Core Principle:** Every reviewer must answer a standard set of questions based on change type. If any question cannot be answered with confidence, the RFC is not approval-ready.

**Framework Components:**
1. **Change Classification** — Emergency, Normal, Standard (existing)
2. **Review Questions by Change Type** — Standardized checklist
3. **Decision Criteria** — Clear yes/no approval gates
4. **Information Requirements** — What must be in the RFC for approval
5. **Connection to Operation Types** — Map 26 operation types to review criteria

## Change Classification & Risk Levels

| Change Type | Risk Level | Approval Path | CAB Required | Auto-Approval |
|---|---|---|---|---|
| **Emergency** | HIGH | Hierarchical (Requestor → Area Lead) | No | ❌ |
| **Normal** | MINIMAL | RFC Catalog Rules | Yes | ❌ |
| **Standard** | LOW | Defined in RFC Catalog | Yes | ✅ |

**Definitions:**

**Emergency (High Risk):**
- Production incidents requiring immediate fix
- Security patches or critical vulnerabilities
- Hierarchical approval only (no CAB review)
- Bypass standard review process with documented justification
- Post-incident review required

**Normal (Minimal Risk):**
- Planned infrastructure or configuration changes
- Routine maintenance or updates
- CAB approval via RFC Catalog articles
- Requires independent reviewer sign-off

**Standard (Low Risk):**
- Regular operational changes covered in catalog
- Auto-approval possible if RFC Catalog article exists
- Independent reviewer verification of catalog compliance
- Reduced review overhead

### 3. Segregation of Duties — Deep Dive (From Vera's Feedback)

**Key Principle:** Reviewers MUST come from outside the Value Stream proposing the change.

**Why This Matters:**

Segregation of duties is a mandatory control in change management governance. It prevents conflicts of interest where:
- A team cannot approve their own changes (risk of bypassing controls)
- Accountability is independent (reviewer has no vested interest in the change outcome)
- Compliance violations are caught by external eyes (auditable, transparent)

**Enforcement Mechanism:**

When an RFC is submitted:
1. **Extract requesting team/Value Stream** from RFC metadata
2. **Identify available reviewers** from other Value Streams (not the requesting one)
3. **Auto-assign reviewer** from approved list for that risk tier
4. **Reject RFC if:** No independent reviewer available (escalate to CAB Chair)
5. **Audit rule:** Monthly verification that 100% of RFCs have segregation of duties compliant

**Impact on Reviewer Assignment:**

| Scenario | Action | Reason |
|---|---|---|
| Requester: Platform VS, Reviewer: Database VS | ✅ APPROVE | Independent; different teams |
| Requester: Platform VS, Reviewer: Platform VS (different team) | ⚠️ ASSESS | Same Value Stream; need additional controls |
| Requester: Platform VS, Reviewer: Platform VS (same team lead) | ❌ REJECT | Same team; violates segregation |
| Requester: Platform VS, Reviewer: Infrastructure VS | ✅ APPROVE | Independent; different Value Stream |

**Automation Rule (Phase 1):**
```
IF requester.team == reviewer.team THEN REJECT
ELSE IF requester.value_stream == reviewer.value_stream THEN ESCALATE_TO_TIER2
ELSE APPROVE_ASSIGNMENT
```

**Related Controls:**
- Independent reviewer must not be a direct report of the requesting team lead
- Reviewer cannot override segregation rule (immutable policy)
- Post-implementation validation is also done by different reviewer (if possible)

## Review Questions Framework

### Core Review Questions (All Change Types)

Every reviewer must answer these questions before approval, regardless of change type:

| # | Question | Purpose | Reject if... |
|---|---|---|---|
| **Q1** | Is segregation of duties satisfied? | Governance | Reviewer is from requesting team |
| **Q2** | What is the business justification? | Context | No clear reason for the change |
| **Q3** | What is the blast radius if this fails? | Risk | Impact unclear or too broad for change type |
| **Q4** | Is there a rollback plan? | Safety | No rollback or recovery procedure |
| **Q5** | What testing was done? | Quality | No evidence of validation |
| **Q6** | When is the change window? | Timing | Conflicts with freeze periods or high-traffic |

**Decision Rule:** If any core question cannot be answered, the RFC is **NOT approval-ready**. Request modifications or reject.

---

### Emergency Changes — Review Checklist

**Context:** Production incidents requiring immediate fix, security patches, critical vulnerabilities.

**Review Questions:**

| # | Question | What to Look For | Red Flags |
|---|---|---|---|
| **E1** | Is this truly an emergency? | Active incident ticket, SEV classification | No incident reference, planned work disguised as emergency |
| **E2** | What is the immediate risk if we don't deploy? | Customer impact, data loss, security exposure | Vague impact, "nice to have" labeled as urgent |
| **E3** | Has Area Lead approved? | Hierarchical approval in place | No Area Lead sign-off |
| **E4** | What is the minimal change scope? | Smallest possible fix to restore service | Scope creep, bundled changes |
| **E5** | What is the rollback procedure? | One-step rollback, tested path | Complex rollback, untested procedure |
| **E6** | Is post-incident review scheduled? | Follow-up meeting planned | No commitment to post-review |

**Approval Criteria:**
- ✅ All 6 core questions answered
- ✅ All 6 emergency-specific questions answered
- ✅ Area Lead approval present
- ✅ Incident ticket linked
- ✅ Rollback procedure documented

**Reject if:**
- Not a true emergency (route to Normal track)
- Scope too broad for emergency process
- No rollback plan

---

### Normal Changes — Review Checklist

**Context:** Planned infrastructure or configuration changes, routine maintenance, updates.

**Review Questions:**

| # | Question | What to Look For | Red Flags |
|---|---|---|---|
| **N1** | Which RFC Catalog article applies? | Specific runbook or procedure reference | No catalog article, "custom" change |
| **N2** | What Operation Type is this? | One of 26 documented operation types | Undefined operation type |
| **N3** | What systems/services are affected? | Clear scope: databases, applications, infrastructure | Vague scope like "platform" |
| **N4** | What is the expected downtime? | 0 minutes, < 5 minutes, planned maintenance window | Unplanned downtime, no estimate |
| **N5** | What testing was done in non-prod? | Test environment validation, staging deployment | No testing, "will test in prod" |
| **N6** | What is the rollback complexity? | Simple revert, database restore, config rollback | Irreversible changes, data migrations |
| **N7** | What is the deployment ring? | Ring 0 (internal), Ring 1 (pilot), Ring 3 (prod) | Ring 3 without Ring 0/1 validation |
| **N8** | Who validated the change plan? | Peer review, technical lead sign-off | No technical review |

**Approval Criteria:**
- ✅ All 6 core questions answered
- ✅ All 8 Normal-specific questions answered
- ✅ RFC Catalog article linked
- ✅ Operation Type identified
- ✅ Testing evidence provided (screenshots, test logs)
- ✅ Rollback plan documented

**Reject if:**
- No RFC Catalog article (create one first)
- No testing in non-prod
- Rollback plan missing or unclear
- Ring 3 deployment without Ring 0/1 validation

---

### Standard Changes — Review Checklist

**Context:** Regular operational changes covered in RFC Catalog, low-risk, repeatable procedures.

**Review Questions:**

| # | Question | What to Look For | Red Flags |
|---|---|---|---|
| **S1** | Which RFC Catalog article is being followed? | Exact runbook reference with version | No catalog article, outdated version |
| **S2** | Are all catalog prerequisites met? | Checklist items completed | Prerequisites skipped |
| **S3** | Is this change within catalog parameters? | Scope, timing, systems match catalog | Deviations from catalog |
| **S4** | Has this procedure succeeded before? | Historical success rate > 95% | First-time use, high failure history |
| **S5** | Is automation being used? | Automated deployment script/tool | Manual execution (should be automated) |

**Approval Criteria:**
- ✅ All 6 core questions answered
- ✅ All 5 Standard-specific questions answered
- ✅ RFC Catalog article compliance verified
- ✅ Automation in use (if catalog requires it)
- ✅ Prerequisites checklist complete

**Auto-Approval Eligible if:**
- Catalog article exists and is current
- All prerequisites met
- Historical success rate > 95%
- No deviations from catalog parameters

**Reject if:**
- Catalog article outdated or missing
- Prerequisites not met
- Deviations from catalog (route to Normal track)

## Review Process Steps

**Step 1: RFC Submission (Requester)**
- Requester creates RFC in Jira
- Requester selects change type: Emergency / Normal / Standard
- System assigns matching review checklist based on type (E/N/S questions)
- System auto-assigns independent reviewer (segregation of duties enforced)
- Reviewer receives notification with assigned checklist

**Step 2: Reviewer Evaluation (Manual)**
- Reviewer opens RFC and reviews assigned checklist (E/N/S)
- Answers all core questions (Q1-Q6)
- Answers change-type-specific questions (E1-E6, N1-N8, or S1-S5)
- Documents answers in RFC comments

**Step 3: Decision**
- **Approve:** All questions answered satisfactorily, no red flags
- **Request Modifications:** Missing information or unclear answers, specific questions noted
- **Reject:** Red flags present, does not meet approval criteria, route to correct track

**Step 4: Implementation (if approved)**
- Requester executes change
- Evidence collected (logs, screenshots)
- RFC updated to "Implemented"

**Step 5: Post-Implementation**
- Reviewer validates success
- CFR tracked (did change cause incident?)
- RFC closed or escalated if issues found

## Review Process Flow

```
┌─────────────────────────────────────────────────────────────┐
│  RFC SUBMITTED                                              │
│  • Requester creates RFC in Jira                            │
│  • Selects change type: Emergency / Normal / Standard       │
│  • System assigns review checklist based on type (E/N/S)    │
│  • Auto-assigns independent reviewer (segregation check)    │
└─────────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────────┐
│  REVIEWER EVALUATION                                        │
│  Reviewer answers questions:                                │
│  • Core Questions (Q1-Q6) — all change types                │
│  • Type-Specific Questions:                                 │
│    - Emergency: E1-E6 (6 questions)                         │
│    - Normal: N1-N8 (8 questions)                            │
│    - Standard: S1-S5 (5 questions)                          │
└─────────────────────────────────────────────────────────────┘
                           ↓
                  ┌────────┴────────┐
                  │  ALL QUESTIONS  │
                  │    ANSWERED?    │
                  └────────┬────────┘
                           │
          ┌────────────────┼────────────────┐
          │                │                │
         NO               YES              UNCLEAR
          │                │                │
          ↓                ↓                ↓
┌──────────────────┐ ┌─────────────┐ ┌──────────────────┐
│ REQUEST MODS     │ │  RED FLAGS? │ │ REQUEST MODS     │
│                  │ └──────┬──────┘ │                  │
│ • Document what  │        │        │ • Ask clarifying │
│   is missing     │   ┌────┴────┐   │   questions      │
│ • Return to      │   │   YES   │   │ • Return to      │
│   requester      │   │   or    │   │   requester      │
└──────────────────┘   │   NO?   │   └──────────────────┘
                       └────┬────┘
                            │
                  ┌─────────┴─────────┐
                 YES                 NO
                  │                   │
                  ↓                   ↓
         ┌────────────────┐  ┌────────────────┐
         │     REJECT     │  │    APPROVE     │
         │                │  │                │
         │ • Document     │  │ • Document     │
         │   reason       │  │   decision     │
         │ • Route to     │  │ • Notify       │
         │   correct      │  │   requester    │
         │   track        │  │ • Clear for    │
         └────────────────┘  │   deployment   │
                             └────────┬───────┘
                                      │
                                      ↓
                         ┌────────────────────────┐
                         │   IMPLEMENTATION       │
                         │   Requester executes   │
                         └────────────┬───────────┘
                                      │
                                      ↓
                         ┌────────────────────────┐
                         │ POST-IMPLEMENTATION    │
                         │ • Validate success     │
                         │ • Track CFR            │
                         │ • Close RFC            │
                         └────────────────────────┘
```

**Key Decision Points:**

1. **All questions answered?**
   - NO → Request modifications (missing information)
   - UNCLEAR → Request modifications (clarifying questions needed)
   - YES → Proceed to red flags check

2. **Red flags present?**
   - YES → Reject (document reason, route to correct track)
   - NO → Approve (document decision, clear for deployment)

**Review Time Targets:**
- **Standard:** < 2 hours (simple checklist)
- **Normal:** < 24 hours (detailed review)
- **Emergency:** < 4 hours (fast track with post-review)

**Primary Metrics (DORA Framework):**

  - Measurement: Deployment Failure Reporting project (separate pipeline failures from incidents)

- **Approval Turnaround Time** — Time from RFC submission to approval
  - Target: < 24 hours for Standard/Normal
  - Target: < 4 hours for Emergency (with post-review)

- **Reviewer Compliance** — % of changes with proper segregation of duties
  - Target: 100% compliance
  - Audit: Monthly review of approval logs

- **Change Volume Processed** — Changes approved per reviewer per cycle
  - Baseline: Current state
  - Target: Scale to 300+ changes/month without quality degradation

## Pilot Approach for April 16 Presentation to Vera

**Objective:** Validate review questions framework with real RFCs before full rollout.

**Pilot Scope:**
1. Select top 3 Normal Operation Types by RFC volume (from M2.1 analysis)
2. Apply N1-N8 review questions to last 5 RFCs of each type
3. Measure: How many RFCs would have been approved/rejected/modified under new framework?
4. Collect: What information was missing? What questions were hardest to answer?
5. Refine: Adjust questions based on findings

**Why Top 3 Normal Types:**
- Highest volume = most impact on approval bottlenecks
- Normal (not Standard) = more complex, better test of framework
- Normal (not Emergency) = planned changes, easier to validate testing/rollback

**Pilot Metrics:**
- % of RFCs that are "approval-ready" (all questions answered)
- Average time to answer all review questions
- Most common missing information
- Reviewer confidence score (survey)

**Success Criteria for Pilot:**
- > 70% of RFCs have all questions answerable from RFC content
- Reviewers report higher confidence in decisions
- Clear pattern of missing information identified (informs RFC template updates)

**Presentation Structure (April 16):**
1. **Problem Reframed:** CAB Reviewers lack structured review process (not team structure issue)
2. **Proposed Solution:** Review questions framework by change type
3. **Example Walkthrough:** Show N1-N8 questions applied to real RFC
4. **Pilot Plan:** Top 3 Normal operation types, 15 RFCs total, 2-week timeline
5. **Expected Outcomes:** Faster approvals, consistent decisions, clearer RFC requirements
6. **Ask:** Approval to pilot with current CAB reviewers
