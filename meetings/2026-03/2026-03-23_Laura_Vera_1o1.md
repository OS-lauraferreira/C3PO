##### Operation Types Canonical Model (Complete Analysis)

**The Discovery:**
Instead of managing two separate lists (22 Standard + 26 Normal), analysis reveals ONE unified canonical set of 32 operation types that covers all change patterns from 2025.

**Breakdown:**

| Category | Count | Details |
|---|---|---|
| **Total Canonical Types** | **32** | Complete unique set covering all observed operations |
| **Standard applicability** | 22 types | Subset approved for self-service (18 current + 4 recommended) |
| **Normal applicability** | 26 types | Subset requiring CAB review (18 overlap + 8 unique) |
| **Overlap (both contexts)** | 18 types | Same operation can be Standard OR Normal depending on scope |

**What the overlap means:**
The same operation type (e.g., "Service restart") can be:
- **Standard** if: Single instance, testing ring, < 5 min downtime
- **Normal** if: Multiple instances, production ring, > 5 min downtime, coordinated restart needed

Routing is determined by: operation type + scope metadata (rings, teams, impact).

**Duplicate Runbooks Analysis:**
- Identified runbooks with multiple implementations (legacy + current)
- Resolution: Keep legitimate variants; deprecate legacy debt
- Action: Consolidate during ODC-CAT creation (map 32 types to canonical runbooks)

**Impact for ODC-CAT:**
- ✅ Simplifies from 48 articles → **32 articles**
- ✅ Each article includes: Standard conditions + Normal conditions + Emergency escalation rules
- ✅ Single source of truth per operation (no duplication)
- ✅ Enables deterministic automation in M2.2 (no logic duplication)

**Governance Model:**

##### Reviewers Framework — Executive Summary

**Strategic Goal:** Reduce approval bottlenecks and change failure rates by creating scalable, risk-based reviewer empowerment rules with segregation of duties.

**The Problem We're Solving:**
- No independent reviewer structure (requesters & approvers often overlap)
- Lack of segregation of duties (governance violations)
- Approval bottleneck (changes stuck in review)
- Manual decision-making (inconsistent criteria)
- **Impact:** Delayed deployments, compliance risk, unclear governance

**The Solution: Three-Tier Reviewer Model**

| Tier | Role | Authority | Changes |
|---|---|---|---|
| **Tier 1** | Junior Reviewer | Standard changes (low-risk catalog operations) | Auto-approval enabled |
| **Tier 2** | Senior Reviewer | Normal changes + Ring 3 infrastructure | Independent review required |
| **Tier 3** | Lead Reviewer (CAB Chair) | Emergency changes + final authority | Post-review analysis |

**Key Rules:**
- ✅ **Segregation of Duties:** Reviewers CANNOT be from requesting team (automated enforcement)
- ✅ **Risk Classification:** Changes auto-classified as Emergency/Normal/Standard based on ODC Catalog
- ✅ **Authority Boundaries:** Clear decision rights by tier; escalation rules defined
- ✅ **Metrics-Driven:** CFR < 5%, approval turnaround < 24h, 100% compliance

**Approval Workflow:**
```
Requester submits RFC
  ↓ (Auto-classify by ODC Catalog)
  ↓ Area Lead validates (same team)
  ↓ Independent Reviewer assesses risk (different team)
  ↓ Route to appropriate tier (Tier 1/2/3)
  ↓ Approve / Request modifications / Escalate
  ↓ Implementation + Post-validation
```

**Automation Roadmap (3 Phases):**
- **Phase 1 (Q2 2026):** Auto-classify, auto-assign tier, enforce segregation, send notifications
- **Phase 2 (Q3 2026):** Auto-approve Standard changes, route to Jira boards, generate compliance reports
- **Phase 3 (Q4 2026+):** ML-based risk scoring, workload balancing, post-implementation automation

**Success Metrics:**
- **Change Failure Rate (CFR):** < 5% (currently: baseline unknown)
- **Approval Turnaround:** < 24h for Standard/Normal; < 4h for Emergency
- **Reviewer Compliance:** 100% segregation of duties (no violations)
- **Change Volume:** Scale to 300+ changes/month without quality degradation

##### Reviewers Framework — Resumo Executivo (Português-PT)

**Objetivo Estratégico:** Reduzir os bottlenecks de aprovação e as change failure rate criando regras escaláveis de empoderamento de reviewers baseadas em risco com segregação de deveres.

**O Problema que estamos a resolver:**
- Sem estrutura independente de reviewers (requesters & approvers frequentemente sobrepõem-se)
- Falta de segregação de deveres (violações de governança)
- Bottleneck de aprovação (mudanças presas em revisão)
- Tomada de decisão manual (critérios inconsistentes)
- **Impacto:** Deployments atrasados, risco de conformidade, governança pouco clara

**A Solução: Modelo de Três Níveis de Revisores**

| Nível       | Função                                | Autoridade                                             | Mudanças                         |
| ----------- | ------------------------------------- | ------------------------------------------------------ | -------------------------------- |
| **Nível 1** | Reviewer Júnior                       | Mudanças padrão (operações de catálogo de baixo risco) | Auto-aprovação habilitada        |
| **Nível 2** | Reviewer Sénior                       | Mudanças normais + infraestrutura Ring 3               | Revisão independente obrigatória |
| **Nível 3** | Reviewer Responsável (Presidente CAB) | Mudanças emergenciais + autoridade final               | Análise pós-revisão              |

**Regras-Chave:**
- ✅ **Segregação de Deveres:** Reviewers NÃO podem ser da equipa que requer a mudança (execução automatizada)
- ✅ **Classificação de Risco:** Mudanças auto-classificadas como Emergencial/Normal/Padrão com base no Catálogo ODC
- ✅ **Limites de Autoridade:** Direitos de decisão claros por nível; regras de escalação definidas
- ✅ **Orientado por Métricas:** CFR < 5%, tempo de aprovação < 24h, 100% conformidade

**Fluxo de Aprovação:**
```
Requester submete RFC
  ↓ (Auto-classificar pelo Catálogo ODC)
  ↓ Area Lead valida (mesma equipa)
  ↓ Reviewer Independente avalia risco (equipa diferente)
  ↓ Direcionar para nível apropriado (Nível 1/2/3)
  ↓ Aprovar / Solicitar modificações / Escalar
  ↓ Implementação + Validação pós-implementação
```

**Roteiro de Automação (3 Fases):**
- **Fase 1 (Q2 2026):** Auto-classificar, auto-atribuir nível, executar segregação, enviar notificações
- **Fase 2 (Q3 2026):** Auto-aprovar mudanças padrão, direcionar para boards Jira, gerar relatórios de conformidade
- **Fase 3 (Q4 2026+):** Pontuação de risco baseada em ML, balanceamento de carga de trabalho, automação pós-implementação

**Métricas de Sucesso:**
- **Taxa de Falha de Mudança (CFR):** < 5% (atual: baseline desconhecido)
- **Tempo de Aprovação:** < 24h para Padrão/Normal; < 4h para Emergencial
- **Conformidade de Reviewers:** 100% segregação de deveres (sem violações)
- **Volume de Mudanças:** Escalar para 300+ mudanças/mês sem degradação de qualidade
