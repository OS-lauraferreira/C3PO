## 2. Framework Overview — Review Questions by Change Type

### **Core Principle**

Every reviewer must answer a **standard set of questions** based on change type. If any question cannot be answered with confidence, the RFC is **not approval-ready**.

### **🎯 Strategic Implementation Approach**

**Phase 1: Define & Control Framework (Manual Process First)**
- Establish review questions framework (E/N/S checklists)
- Train human reviewers on structured evaluation
- Collect baseline data: approval time, decision consistency, missing information patterns
- **Goal:** Stabilize human review process with clear criteria

**Phase 2: Build AI Review Agent (In Parallel or After Phase 1)**
- Develop AI agent that answers same review questions (Q1-Q6, E1-E6, N1-N8, S1-S5)
- Agent reads RFC content and attempts to answer all checklist questions
- Agent flags: "Cannot answer N5 — no testing evidence provided"
- **Goal:** Automated first-pass review to identify missing information

**Phase 3: Human vs AI Comparison**
- Run AI agent in parallel with human reviewers (shadow mode)
- Compare answers: Where do AI and human agree/disagree?
- Validate: Does AI catch same red flags as human reviewers?
- **Goal:** Establish AI agent accuracy baseline

**Phase 4: AI-Assisted Review Workflow**
- AI agent performs first-pass review → flags missing information
- Human reviewer validates AI findings → makes final approval decision
- Benefits: Faster reviews, consistent validation, reduced human review burden

**Why This Approach:**
- ✅ **Human baseline first** — AI learns from established human process, not vice versa
- ✅ **Validation & trust** — Compare AI vs human to build confidence in AI accuracy
- ✅ **Scalability** — AI handles repetitive validation, humans focus on judgment calls
- ✅ **Continuous improvement** — AI agent learns from human reviewer corrections

**Framework Components:**
1. **Core Questions (Q1-Q6)** — All change types
2. **Change-Type-Specific Questions:**
   - Emergency: E1-E6 (6 questions)
   - Normal: N1-N8 (8 questions)
   - Standard: S1-S5 (5 questions)
3. **Decision Criteria:** Approve / Request Modifications / Reject

---

### **Core Review Questions (All Change Types)**

| # | Question | Purpose | Reject if... |
|---|---|---|---|
| **Q1** | Is segregation of duties satisfied? | Governance | Reviewer is from requesting team |
| **Q2** | What is the business justification? | Context | No clear reason for the change |
| **Q3** | What is the blast radius if this fails? | Risk | Impact unclear or too broad for change type |
| **Q4** | Is there a rollback plan? | Safety | No rollback or recovery procedure |
| **Q5** | What testing was done? | Quality | No evidence of validation |
| **Q6** | When is the change window? | Timing | Conflicts with freeze periods or high-traffic |

---

### **Emergency Changes — Review Checklist (E1-E6)**

| # | Question | What to Look For | Red Flags |
|---|---|---|---|
| **E1** | Is this truly an emergency? | Active incident ticket, SEV classification | No incident reference, planned work disguised as emergency |
| **E2** | What is the immediate risk if we don't deploy? | Customer impact, data loss, security exposure | Vague impact, "nice to have" labeled as urgent |
| **E3** | Has Area Lead approved? | Hierarchical approval in place | No Area Lead sign-off |
| **E4** | What is the minimal change scope? | Smallest possible fix to restore service | Scope creep, bundled changes |
| **E5** | What is the rollback procedure? | One-step rollback, tested path | Complex rollback, untested procedure |
| **E6** | Is post-incident review scheduled? | Follow-up meeting planned | No commitment to post-review |

**🔍 Connection to Ops Review Findings:**
- **E1 addresses:** Emergency classification accuracy gap (RDINC-78358, RDINC-78052)
- **E2 addresses:** System-wide impact definition clarity
- **E6 addresses:** Post-mortem commitment to understand root cause

**Approval Criteria:**
- ✅ All 6 core questions (Q1-Q6) answered
- ✅ All 6 emergency-specific questions (E1-E6) answered
- ✅ Area Lead approval present
- ✅ Incident ticket linked
- ✅ Rollback procedure documented

---

### **Normal Changes — Review Checklist (N1-N8)**

| # | Question | What to Look For | Red Flags |
|---|---|---|---|
| **N1** | Which RFC Catalog article applies? | Specific runbook or procedure reference | No catalog article, "custom" change |
| **N2** | What Operation Type is this? | One of 26 documented operation types | Undefined operation type |
| **N3** | What systems/services are affected? | Clear scope: databases, applications, infrastructure | Vague scope like "platform" |
| **N4** | What is the expected downtime? | 0 minutes, < 5 minutes, planned maintenance window | Unplanned downtime, no estimate |
| **N5** | What testing was done in non-prod? | Test environment validation, staging deployment | No testing, "will test in prod" |
| **N6** | What is the rollback complexity? | Simple revert, database restore, config rollback | Irreversible changes, data migrations |
| **N7** | What is the deployment ring? | Ring 0 (internal), Ring 1 (pilot), Ring 3 (prod) | Ring 3 without Ring 0/1 validation |
| **N8** | Who validated the change plan? | Peer review, technical lead sign-off | No technical review |

**🔍 Connection to Ops Review Findings:**
- **N2 addresses:** Normal Planned 76.92% — map to Operation Types to identify automation candidates (feeds M2.3)
- **N1 addresses:** RFC Catalog article enforcement (M2.1 Change Catalog consolidation)
- **N5, N7 address:** Ring distribution inversion (ring=3/ga at 48 executions vs ring=1 at 33)

**Approval Criteria:**
- ✅ All 6 core questions (Q1-Q6) answered
- ✅ All 8 Normal-specific questions (N1-N8) answered
- ✅ RFC Catalog article linked
- ✅ Operation Type identified
- ✅ Testing evidence provided
- ✅ Rollback plan documented

---

### **Standard Changes — Review Checklist (S1-S5)**

| # | Question | What to Look For | Red Flags |
|---|---|---|---|
| **S1** | Which RFC Catalog article is being followed? | Exact runbook reference with version | No catalog article, outdated version |
| **S2** | Are all catalog prerequisites met? | Checklist items completed | Prerequisites skipped |
| **S3** | Is this change within catalog parameters? | Scope, timing, systems match catalog | Deviations from catalog |
| **S4** | Has this procedure succeeded before? | Historical success rate > 95% | First-time use, high failure history |
| **S5** | Is automation being used? | Automated deployment script/tool | Manual execution (should be automated) |

**Auto-Approval Eligible if:**
- Catalog article exists and is current
- All prerequisites met
- Historical success rate > 95%
- No deviations from catalog parameters
